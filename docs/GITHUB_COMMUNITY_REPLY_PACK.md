# GitHub Community Reply Pack

Use these replies only when someone is asking about first pull requests, beginner open source, good first issues, or GitHub workflow help.

Do not paste these into unrelated threads. The goal is to help first and share the repo only when it fits.

## Reply: How Do I Start Open Source?

```md
The easiest path is to pick a small issue, comment before starting, make one focused change, and open a PR with proof of testing.

If you want a small practice repo for the full GitHub flow, I maintain Open Source Starter Lab:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab

It has good first issues, skill-based first issues, CI checks, and a Discussion where you can say what you know and get a suggested issue:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/discussions/35

The main thing is to start small. Docs, tests, examples, and Git workflow fixes are real open-source contributions too.
```

## Reply: I Know JavaScript, Where Do I Start?

```md
If you know JavaScript, start with a small issue where you can read the code and change one behavior or one output message.

I made a JavaScript-friendly first issue here:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/issues/29

The repo is small, uses a TypeScript CLI, and has `npm run check` so you can practice a real PR workflow without a huge codebase.
```

## Reply: I Know Python But The Repo Uses TypeScript

```md
You can still contribute even if your main language is Python. A good first step is docs, examples, setup guides, or explaining the workflow for other Python learners.

I made a Python learner first issue here:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/issues/30

That kind of contribution is useful because many beginners need help connecting what they already know to GitHub workflow.
```

## Reply: I Am Scared To Annoy Maintainers

```md
That is normal. A good way to avoid wasting maintainer time is:

1. Pick a small issue.
2. Comment before starting.
3. Make one focused change.
4. Run the project checks.
5. Explain what changed and how you tested it.

I built a beginner practice repo around exactly that workflow:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab

You can also ask for an issue suggestion here:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/discussions/35
```

## Reply: Good First Issues Are Already Taken

```md
That happens a lot. Look for active repos where maintainers reply, issues have clear acceptance criteria, and contributors are asked to comment before starting.

I keep a small repo with skill-based first issues here:

https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/blob/main/docs/SKILL_BASED_FIRST_ISSUES.md

You can choose by what you know: HTML/CSS, JavaScript, Python, docs, testing, or Git.
```

## Maintenance Rule

If someone replies, answer like a maintainer:

- be specific
- suggest one issue
- explain why it fits
- avoid asking for stars
- avoid posting the same reply everywhere
