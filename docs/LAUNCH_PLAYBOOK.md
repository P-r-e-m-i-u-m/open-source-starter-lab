# Launch Playbook

Use this when launching Open Source Starter Lab, sharing it with beginners, or inviting people to make their first pull request.

The goal is simple: help people start. Do not spam, do not ask for fake stars, and do not promise outcomes the repo cannot control.

## Product Positioning

Open Source Starter Lab is a beginner-friendly GitHub lab for people who want to make their first pull request, first open-source contribution, or first good first issue without getting lost.

It gives contributors:

- a live website
- clear good first issues
- skill-based issue routes
- a weekly assignment thread
- copy-paste setup commands
- CI checks
- PR welcome guidance
- a First Merge Wall
- a second PR path after the first merge

## Comment Replies

When someone says they are new:

```md
Welcome. Start small and do not worry about making it perfect.

Pick one unassigned good first issue, comment that you want to work on it, then run:

`npm run check`

If you are not sure which issue fits you, reply in the weekly thread with your skill and available time:
https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/discussions/44
```

When someone asks for an issue:

```md
Done @username.

Start with:
`npm run check`

Keep the PR small. Change one thing, then add your check output in the PR.
```

When someone opens a PR:

```md
Thanks for opening this. Please keep the PR focused on one change.

Before review, add:

- what changed
- why it helps
- `npm run check` output

That makes it easier to review and merge cleanly.
```

After merge:

```md
Merged. Nice work.

You can add your first merge to the First Merge Wall:
https://github.com/P-r-e-m-i-u-m/open-source-starter-lab/blob/main/docs/FIRST_MERGE_WALL.md
```

## Seven-Day Distribution Plan

### Day 1: Launch

- Share the LinkedIn post.
- Share the X post.
- Pin the website link in the repo description.
- Reply fast to every GitHub comment.

### Day 2: Community Help

- Answer beginner questions in GitHub Discussions.
- Share the Reddit post only where project sharing is allowed.
- Invite 3-5 people who are already learning GitHub.

### Day 3: Proof

- Share one screenshot of the website or a clean PR flow.
- Mention one real beginner problem the repo solves.
- Do not ask for stars. Ask people to try one issue.

### Day 4: Article

- Publish the DEV or Hashnode article.
- Link to the website and Start Here guide.
- Add a short "who this is for" section.

### Day 5: Maintainer Energy

- Review open PRs.
- Thank contributors.
- Add merged contributors to the First Merge Wall.
- Suggest second PR paths.

### Day 6: Search And Replies

- Search GitHub, Reddit, DEV, and Hashnode for people asking about first PRs.
- Reply only when your repo is directly useful.
- Keep replies short and helpful.

### Day 7: Recap

- Post a weekly recap:
  - contributors helped
  - PRs merged
  - issues still open
  - best first tasks for next week

## Spread Loop

The repo should guide people through this loop:

1. Social post or website link.
2. Website explains the repo in seconds.
3. Contributor opens Start Here.
4. Contributor picks one issue or asks in the weekly thread.
5. PR Welcome Guard tells them what proof to include.
6. Merge adds visible proof in First Merge Wall.
7. Contributor gets a second PR suggestion.
8. Contributor shares their first merge, bringing the next beginner.

## Metrics To Watch

Good launch signals:

- issue comments from new people
- weekly discussion replies
- first-time PRs
- PRs with `npm run check` output
- merged first PRs
- contributors returning for a second PR

Weak signals:

- stars with no issue comments
- big PRs that are hard to review
- copied comments across unrelated repos
- people asking what the repo does after opening it

## What Not To Do

- Do not spam big repositories.
- Do not ask for fake stars or forks.
- Do not promise jobs, badges, or guaranteed merges.
- Do not open meaningless PRs in other projects just to advertise.
- Do not over-automate replies where a human answer is needed.

Real trust comes from useful issues, fast review, kind replies, and visible proof.
